# 챕터 02 — 잠재적 문제 선제 보고 규칙

## 원칙
코드를 수정하거나 새로 짤 때, 조금이라도 문제가 예상되면 **사용자가 묻기 전에 먼저 말한다.**

## 언제 보고하는가

### 1. 항상 보고해야 하는 상황
- 데이터 누락 가능성 (API가 특정 연도/기업 데이터를 반환하지 않을 수 있을 때)
- API 응답 형식 가정 (응답 구조가 바뀌면 코드가 깨질 수 있을 때)
- 엣지케이스 (빈 리스트, None, 0값, 음수 등)
- 과거/미래 연도 처리 차이
- 단순 버그 수정이라도 사이드이펙트가 보이면 함께 언급

### 2. 보고 형식
```
⚠️ 잠재적 문제: [문제 설명]
- 원인: [왜 이 문제가 생길 수 있는가]
- 영향: [어떤 상황에서 발생하는가]
- 대응 방안: [권장 해결책]
```

### 3. 사이드이펙트 체크
코드 수정 시 아래를 확인한다:
- 해당 함수를 호출하는 다른 함수가 있는가?
- 반환값의 형식이 바뀌었는가?
- 전역 변수/공유 상태에 영향을 주는가?
- Google Sheets 쓰기 작업에서 기존 데이터를 덮어쓸 위험이 있는가?

## 체크리스트
- [ ] API 응답이 예상과 다를 경우를 처리했는가?
- [ ] None/빈값 처리를 했는가?
- [ ] 수정한 함수의 호출부를 모두 확인했는가?
- [ ] 사이드이펙트를 사용자에게 미리 보고했는가?
